# План реализации zibby-service до release

## 1) Цель и границы

Данный план описывает поэтапную реализацию требований ТЗ для выпуска production-версии `zibby-service` под Windows и Linux с полной автосборкой, тестами, проверками безопасности, генерацией документации и установщиками.

## 2) Дорожная карта (этапы)

### Этап A — Базовая инженерная готовность (Build/Test/Package)

1. Утвердить CMake-опции (`BUILD_TESTS`, `BUILD_CALLS`, `BUILD_PLUGINS`, `USE_STATIC_LIBS`).
2. Настроить генерацию артефактов CPack для платформ:
   - Windows: `NSIS`, `ZIP`.
   - Linux: `DEB`, `RPM`, `TGZ`.
3. Выводить все артефакты в каталог `installers/`.
4. Генерировать файл `SHA256SUMS.txt` после упаковки.
5. Добавить `build.sh`, `build.ps1`, `build.bat` с режимами:
   - полная сборка;
   - только тесты (`--test-only` / `-TestOnly`).
6. В сборочных скриптах выполнять тесты до упаковки (обязательный gate).

### Этап B — Качество кода и безопасность

1. Включить в CI:
   - линтеры (`cppcheck`, `clang-tidy`);
   - сборку и тесты на Linux и Windows;
   - сборку установщиков и публикацию артефактов.
2. Добавить отдельный workflow CodeQL с регулярным расписанием и анализом PR.
3. Ввести политику блокировки merge при падении линтеров/тестов.
4. Добавить отчет о найденных уязвимостях в процесс релиза.

### Этап C — Core функциональность мессенджера

1. Аккаунт:
   - автосоздание профиля;
   - валидация имени (regex, trim);
   - генерация пары ключей (ECC/RSA).
2. Чаты:
   - личные, группы, каналы;
   - права владельца/модератора;
   - публичность/приватность.
3. Сообщения:
   - отправка, редактирование, удаление;
   - реакции, пересылка;
   - статусы доставки/прочтения.
4. SQLite/SQLCipher:
   - хранение профиля, чатов, истории, ключей;
   - миграции схемы БД.

### Этап D — Сеть, шифрование, устойчивость

1. Peer discovery:
   - multicast/broadcast UDP;
   - кеш найденных peer и ручное добавление IP.
2. Транспорт:
   - TCP для сообщений/сигналинга;
   - UDP для потоков/файлов.
3. Шифрование:
   - обмен ключами (ECDH);
   - сеансовый ключ (AES-GCM/ChaCha20-Poly1305);
   - подпись пакетов и anti-replay.
4. Надежность:
   - crash handler + crash report;
   - single-instance + IPC для CLI;
   - корректный restart через сервис-менеджер ОС.

### Этап E — Медиа и звонки

1. MVP звонков (audio/video/screen-share) на WebRTC/GStreamer.
2. Сигналинг SDP/ICE через существующий TCP-канал.
3. Адаптивный битрейт, завершение звонка при потере P2P.
4. Ограничения CLI: корректные сообщения о недоступности медиа-функций.

### Этап F — API для фронтендов

1. Локальный API (localhost TCP/WebSocket).
2. Аутентификация фронтенда токеном.
3. Асинхронные команды + события (new message, read status, presence).
4. Поддержка нескольких фронтендов одновременно.

### Этап G — Плагины

1. ABI-интерфейс плагинов (`.dll`/`.so`) и `registerPlugin`.
2. Загрузка из папки `plugins/`.
3. CLI-команды: `plugin list|enable|disable|load`.
4. Документация по SDK плагинов и пример плагина.

### Этап H — Сервисные функции и обновления

1. Проверка обновлений с подписью релиза.
2. Автообновление с безопасным rollback.
3. Автозапуск:
   - Windows Service / Task Scheduler;
   - Linux systemd (user/system).

### Этап I — Документация и сопровождение

1. Полное покрытие API-комментариями Doxygen на русском языке:
   - назначение;
   - параметры;
   - возвращаемые значения;
   - исключения;
   - потокобезопасность;
   - пример использования.
2. Генерация HTML-документации в `docs/html`.
3. Обновление `docs/ru` и `docs/en` по каждому модулю.

## 3) Definition of Done (release)

Release считается готовым, когда выполнены все пункты:

1. На Linux и Windows CI зеленый (линтеры, тесты, сборка, упаковка, CodeQL).
2. В `installers/` формируются артефакты платформ и `SHA256SUMS.txt`.
3. Unit и integration тесты запускаются локально и в CI.
4. Документация сгенерирована и включена в поставку.
5. Сервис стартует как демон, CLI подключается к существующему процессу.
6. Реализованы сообщения, статусы прочтения, редактирование, шифрование, звонки (MVP), API, плагины.

## 4) Текущее состояние (на дату обновления плана)

1. Базовая CMake-сборка и простые тесты доступны.
2. Добавлены CI для сборки/линтинга и отдельный CodeQL workflow.
3. Добавлена генерация контрольных сумм `SHA256SUMS.txt`.
4. Артефакты упаковки направляются в каталог `installers/`.
