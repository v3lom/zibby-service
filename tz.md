
После компиляции проекта должны быть сформированы установщики для Windows и Linux, содержащие все необходимые файлы для развертывания сервиса zibby-service. Проект использует CMake с генерацией установщиков через CPack. В репозитории предусмотрены скрипты сборки с TUI для выбора опций и автоматической установки зависимостей. Весь код снабжен подробными комментариями на русском языке. Документация представлена в виде структурированных Markdown-файлов и сгенерированных HTML-страниц.
1. Артефакты после компиляции
-Windows: установщик .exe (NSIS или CPack), содержащий бинарный файл zibby-service.exe, библиотеки, конфигурацию по умолчанию, скрипты для регистрации службы Windows и иконку трея. Также создается портативная версия .zip.
-Linux: пакеты .deb, .rpm и архив .tar.gz. В пакетах прописаны зависимости, создается systemd user service (или системный), добавляется символическая ссылка в /usr/bin.
-Все артефакты помещаются в каталог installers/ после сборки.
2. Система сборки и автоматизация
-Корневой CMakeLists.txt определяет цели, зависимости, опции (BUILD_CALLS, BUILD_PLUGINS, USE_STATIC_LIBS и т.д.).
-CPack настроен на создание установщиков для целевых платформ.
-В корне лежат скрипты:
     build.sh (Linux/macOS) и build.bat/build.ps1 (Windows).
-Скрипты определяют наличие cmake, компилятора, зависимостей. При необходимости предлагают установить недостающие пакеты через системный менеджер (apt, pacman, yum, winget, choco, msys2).
-TUI реализован с помощью dialog/whiptail (Linux) и Windows Forms/PowerShell (Windows) или с использованием простого текстового меню с выбором стрелками. Пользователь выбирает тип сборки, дополнительные компоненты, режим установки зависимостей.
-После конфигурации CMake запускается сборка (make или msbuild) и создание установщиков.
-CI/CD (GitHub Actions) настроен на автоматическую сборку при пуше и создание релизов с установщиками при тегировании.
Общие сведения
Разрабатывается программный продукт zibby-service. Это кроссплатформенный фоновый сервис (демон) для IP-мессенджера. Реализация на C++ с использованием библиотеки Boost. Поддерживаемые ОС: Windows (в виде службы или приложения в трее) и Linux (systemd-сервис). Продукт предоставляет backend для мессенджера: управление аккаунтом, чатами, сообщениями, медиа, звонками, а также CLI для управления и отдельный API для фронтенд-клиентов. Сервис работает в фоне, запускается автоматически при старте системы, поддерживает автообновление из публичного репозитория.
Цель
Создать децентрализованный защищенный мессенджер, работающий напрямую по IP (peer-to-peer) с возможностью подключения к серверным чатам (каналы/группы). Обеспечить полное управление через CLI и возможность подключения внешних фронтендов. Обеспечить шифрование данных, кеширование, локальное хранение, высокую производительность и устойчивость к сбоям.
Функциональные требования
1. Управление аккаунтом пользователя
   У каждого пользователя есть личное пространство. Аккаунт создается автоматически при первом запуске.
   Поля аккаунта:
-Имя учетной записи (отображаемое имя). Если не задано – используется hostname/username системы.
-Аватарка (опционально, файл изображения, не более 200 МБ).
-Описание (опционально, до 2000 символов).
-Дата рождения (опционально).
-Статус: активный, скрыт из поиска, неактивный.
-Публичный ключ шифрования (генерируется при создании).
     Имя аккаунта должно проходить проверку регулярным выражением: не пустое, допустимые символы (определенный набор), без пробелов в начале/конце.
     Пользователь может изменить свои данные через CLI или API. При изменении имени проверка уникальности в пределах известных клиентов не требуется – имя не обязано быть глобально уникальным, идентификация идет по ключам и IP:порту.
2. Чаты, каналы и группы
   Поддерживаются три типа коллективных пространств:
-Личный чат (диалог между двумя пользователями).
-Группа (много пользователей, каждый может писать).
-Канал (владелец публикует сообщения, подписчики только читают, если не разрешено комментирование – но в MVP владелец может разрешить комментарии).
     Для групп и каналов:
-Название, описание (до 2000 символов), аватарка (до 200 МБ).
-Владелец (создатель) имеет права на изменение свойств, удаление, назначение модераторов (для групп).
-Каждый чат имеет уникальный идентификатор на основе ключей участников и/или идентификатора канала.
-Чат может быть публичным (виден в поиске) или приватным (доступ только по прямой ссылке/ключу).
-Участник может скрыть чат из своего интерфейса, но не удалять историю, если иное не указано.
3. Сообщения и медиа
   Поддерживаются типы сообщений:
-Текстовое сообщение (не пустое, до 4000 символов, можно задать ограничение).
-Реакции на сообщение (эмодзи).
-Редактирование сообщения (только автор или владелец канала для своих сообщений; в группах – автор).
-Удаление сообщения (для всех или только у себя – по выбору).
-Пересылка сообщений.
-Отправка изображений, файлов (любого типа, размер до 2 ГБ на передачу, но с предупреждением о лимите; фактический лимит настраивается).
-Голосовые сообщения (аудио, запись через микрофон).
-Стикеры, GIF (поддержка внешних источников или встроенные наборы).
-Отправка эмодзи.
     Файлы и медиа:
-Передача по UDP/TCP с возможностью возобновления.
-При получении файл сохраняется в папку загрузок (по умолчанию: для Windows – %USERPROFILE%\Downloads\zibby-downloads, для Linux – ~/Downloads/zibby-downloads, или в настраиваемую директорию).
-Кеширование медиафайлов (изображения, голосовые сообщения) в локальном хранилище сервиса (например, в appdata/zibby/cache).
-История сообщений хранится в SQLite, зашифрована.
4. Звонки
   Поддерживаются:
-Голосовые звонки (P2P, с шифрованием).
-Видеозвонки.
-Демонстрация экрана.
     Звонок инициируется одним участником, завершается, если инициатор выходит (т.к. нет сервера звонков, звонок P2P). При разрыве соединения звонок завершается. Во время звонка передаются аудио/видеопотоки с адаптивным битрейтом.
5. Поиск и обнаружение
   Сервис периодически сканирует локальные сети (все доступные IP-интерфейсы, включая виртуальные адаптеры, VPN, мосты) на наличие других экземпляров zibby-service.
   Каждый сервис публикует свое присутствие через multicast или широковещательные сообщения (UDP) на определенном порту. В ответе содержатся: публичный IP, порт, hostname, имя пользователя, версия, список доступных IP (все интерфейсы), MAC-адреса, ОС, локальное время, язык, регион.
   Можно добавить кастомные IP и порты вручную через настройки. При обнаружении другого клиента можно начать диалог или добавить в контакты.
   Возможность скрыть себя из поиска (статус "скрыт").
   При подключении к удаленному клиенту (не в локальной сети) требуется знать его IP и порт, а также публичный ключ для установления защищенного соединения.
   Сервис также может обнаруживать другие сервисы (RTSP, SMB, FTP) и предлагать подключение, но для подключения к ним требуется аутентификация (логин/пароль) – это опционально.
6. Управление через CLI
   Сервис предоставляет интерфейс командной строки для управления. Запуск:
-zibby-service --cli – подключиться к уже запущенному сервису (если запущен) и войти в интерактивную консоль.
-Если сервис не запущен, запускается и затем открывается CLI.
-CLI поддерживает как аргументы командной строки (zibby-service --send --user "name" --message "hello"), так и интерактивный режим с TUI (текстовый интерфейс с использованием клавиш стрелок, цветов, черточек).
     В CLI доступны команды:
-Просмотр и изменение своего профиля.
-Просмотр списка контактов, групп, каналов.
-Поиск пользователей в сети (локальной и по заданным IP).
-Отправка текстовых сообщений, файлов (для файлов, голосовых, звонков – выводить сообщение о невозможности в CLI).
-Создание группы/канала, приглашение участников.
-Управление настройками шифрования, сетевыми интерфейсами.
-Очистка локальной истории (с подтверждением).
-Просмотр размера базы данных.
-Управление плагинами (загрузка, активация, деактивация).
-Просмотр логов (с фильтрацией).
-Получение информации о сервисе (версия, uptime, статус).
-Выключение/перезапуск сервиса.
     CLI должен работать без прав администратора/root.
7. API для фронтенда
   Сервис предоставляет локальный API (например, через сокеты, named pipe или HTTP на localhost) для взаимодействия с внешними фронтенд-приложениями.
   API должен обеспечивать весь функционал мессенджера: аутентификация фронтенда (по токену или через подтверждение в CLI), отправка и получение сообщений, управление чатами, передача файлов, уведомления о событиях.
   Каждый фронтенд при подключении сообщает свой идентификатор (название и ссылку на источник). Эта информация может транслироваться в сети как "подключенный фронтенд" (по желанию).
   Несколько фронтендов могут быть подключены одновременно. API должен быть защищен от несанкционированного доступа.
8. Настройки и конфигурация
   Файл конфигурации (например, JSON или TOML) в директории сервиса.
   Настраиваемые параметры:
-Сетевые интерфейсы: какие использовать (все или перечисленные), порты (основной и для звонков), протоколы (UDP/TCP).
-Публичные и приватные ключи (пользователь может задать свои, иначе генерируются).
-Режим производительности (экономия ресурсов или высокая производительность).
-Директории для хранения данных, кеша, загрузок.
-Автозапуск (вкл/выкл).
-Проверка обновлений (вкл/выкл, интервал).
-Логирование (уровень детализации, ротация).
-Настройки шифрования (алгоритм, размер ключа).
Требования к архитектуре и технологиям
1. Язык и библиотеки: C++17/20, Boost.Asio для сетевого взаимодействия, Boost.Process, Boost.Filesystem, Boost.Serialization (при необходимости), SQLite3, OpenSSL (для шифрования), аудио/видео – возможно использование WebRTC или аналогичных библиотек, либо реализация на основе Boost и библиотек кодеки (Opus, VP8 и т.п.). Для кроссплатформенности – CMake.
2. Многопоточность: сервис использует пул потоков для обработки сетевых событий, ввода-вывода, шифрования, файловых операций. Отдельные потоки для звонков.
3. Хранение данных: SQLite с шифрованием (SQLCipher или аналоги). Ключ шифрования базы данных формируется на основе MAC-адреса и мастер-ключа пользователя. Все чувствительные данные (сообщения, история) хранятся в зашифрованном виде. Медиафайлы в кеше могут быть зашифрованы или храниться в открытом виде, но доступ к ним ограничен правами ОС.
4. Сетевое взаимодействие:
-Обнаружение peers: multicast или broadcast UDP на определенном порту.
-Основной протокол обмена сообщениями – поверх TCP (для надежности) и UDP (для потоковых данных, файлов, звонков). Поддержка обхода NAT (UPnP, NAT-PMP) опционально.
-Каждое сообщение подписывается и шифруется асимметрично (публичный ключ получателя) или симметричным сеансовым ключом.
-Для групповых чатов используется шифрование с помощью общего ключа группы, распространяемого между участниками.
-Передача файлов: возможность использования отдельных соединений, фрагментация, контроль целостности.
5. Управление сервисом:
-На Windows: сервис Windows (может работать как консольное приложение для отладки, но в production – как служба). Возможность запуска из трея (иконка в системном трее) для управления автозапуском и завершения.
-На Linux: systemd unit, возможность запуска от пользователя (user service) без root.
-Сервис должен быть единственным экземпляром. При повторном запуске (например, через CLI) должен подключаться к уже работающему экземпляру (через IPC) и передавать команды.
Требования к безопасности и шифрованию
1. Каждый пользователь генерирует пару ключей (RSA или ECC) при первом запуске. Публичный ключ доступен для других пользователей.
2. Для каждого личного чата генерируется временный симметричный ключ (или используется протокол Диффи-Хеллмана) для шифрования сообщений.
3. Групповые чаты: ключ группы распространяется через зашифрованные каналы между участниками.
4. Хранение локальной базы данных: шифрование с использованием ключа, полученного из MAC-адреса и мастер-пароля (или привязанного к оборудованию). Возможно использование системного хранилища ключей (Windows Credential Manager, GNOME Keyring, KWallet) для мастер-ключа.
5. Все сетевые пакеты шифруются. Запрещена передача открытого текста.
6. Защита от атак типа "человек посередине": проверка отпечатков публичных ключей при установлении соединения. Пользователь может вручную подтвердить ключ собеседника (через CLI или фронтенд).
7. При отправке файлов используется шифрование тем же сеансовым ключом, что и сообщения, либо отдельным ключом.
8. При обновлении сервиса проверяется подпись обновления (например, GPG) для предотвращения распространения вредоносного кода.
Требования к хранению данных
1. Директории:
-Для Windows: %APPDATA%\Zibby\ (данные, база, кеш), %USERPROFILE%\Downloads\ZibbyDownloads (загрузки).
-Для Linux: ~/.local/share/zibby/ (данные), ~/.cache/zibby/ (кеш), ~/Downloads/zibby-downloads (загрузки).
-Возможность переопределить пути через конфигурацию.
2. База данных SQLite хранит:
-Профиль пользователя (имя, ключи, настройки).
-Список контактов (с публичными ключами, последним IP/портом, статусом).
-История сообщений по чатам (с указанием чата, автора, времени, типа, содержимого – зашифровано).
-Информацию о группах/каналах (участники, права, ключи).
3. Кеш медиафайлов: файлы хранятся с именами, соответствующими хэшу содержимого, метаданные в базе. Размер кеша может быть ограничен (настраивается). При достижении лимита удаляются старые файлы.
Требования к сервису (демон)
1. Сервис запускается при старте системы, если автозапуск включен. После запуска проверяет наличие обновлений в публичном репозитории (GitHub). При наличии новой версии скачивает, проверяет подпись и устанавливает (с перезапуском). Обновление должно быть безопасным, с сохранением данных пользователя.
2. Сервис должен быть устойчив к сбоям: при необработанном исключении должен создаваться crash report (дамп, стек вызовов) и сохраняться в папку логов. После этого сервис должен быть перезапущен (если это возможно). В случае фатальной ошибки – завершиться, а планировщик ОС (или менеджер служб) должен его перезапустить.
3. Сервис должен предоставлять интерфейс для получения статуса и управления через именованный канал, сокет или DBus (для Linux) – используется CLI для подключения.
4. Режимы работы: производительный (по умолчанию) и экономия ресурсов (снижение частоты сканирования сети, уменьшение количества потоков). Переключение через конфигурацию.
Требования к CLI
1. CLI может быть запущен с аргументами: --cli – вход в интерактивный режим; --send --to "user" --message "text" – отправка сообщения (сервис должен быть запущен); --help – справка.
2. Интерактивный режим (TUI) поддерживает:
-Отображение списка чатов с непрочитанными.
-Просмотр сообщений в выбранном чате.
-Ввод текста, отправка.
-Возможность прикрепления файлов (через диалог выбора файла).
-Управление стрелками, Enter, Esc, комбинации Ctrl+?.
-Использование цветов для различения отправителей, системных сообщений.
3. Неподдерживаемые в CLI действия (звонки, голосовые сообщения) должны отображать сообщение о невозможности и предложить использовать фронтенд.
Требования к API для фронтенда
1. API реализуется через локальный TCP-сокет или WebSocket на localhost: случайный порт (или фиксированный, но с аутентификацией). Первое сообщение от фронтенда – аутентификация (токен, который выводится в CLI при первом подключении или генерируется пользователем). После аутентификации фронтенд может подписываться на события (новые сообщения, статусы).
2. API должен предоставлять методы:
-Получение профиля.
-Получение списка контактов, чатов.
-Отправка сообщений (текст, файл, реакция).
-Получение истории чата.
-Создание группы/канала.
-Управление участниками.
-Передача файлов (с возможностью возобновления).
-Инициация звонка (передача SDP и ICE кандидатов для WebRTC или собственного протокола).
3. Все данные передаются в формате JSON или Protobuf. Команды асинхронны, ответы содержат идентификаторы запросов.
Требования к сетевому взаимодействию
1. Сервис слушает на всех доступных IP-адресах указанный порт (основной) и дополнительный порт для звонков (если нужно). Порт можно изменить в настройках.
2. Для обнаружения использует multicast адрес (например, 239.0.0.1:9876) или широковещательный UDP. Периодически отправляет присутствие и слушает ответы.
3. При обнаружении другого клиента сохраняет его в кеш: IP, порт, публичный ключ, имя. При необходимости инициирует установление прямого TCP-соединения.
4. Поддерживает работу с несколькими сетевыми интерфейсами: отправка исходящих пакетов через тот интерфейс, который соответствует маршруту, либо можно принудительно задать исходящий интерфейс.
5. При подключении к удаленному клиенту (не в локальной сети) используется STUN/TURN (опционально) для обхода NAT. Для упрощения MVP может быть реализован обход через UPnP.
6. Все сообщения имеют уникальные идентификаторы (UUID) и временные метки. Дубликаты отбрасываются.
Требования к мультимедиа (звонки, файлы)
1. Звонки реализованы на основе протокола реального времени (WebRTC или собственный на RTP). Для MVP достаточно аудио (Opus), видео (VP8/H.264) и демонстрации экрана. Используются библиотеки типа libwebrtc или GStreamer с интеграцией в Boost.Asio.
2. При инициации звонка клиенты обмениваются SDP через существующий канал (TCP), затем устанавливают прямое UDP-соединение для медиапотоков.
3. Демонстрация экрана возможна только при наличии графического окружения; при отсутствии – отказ.
4. Файлы передаются по отдельному каналу (UDP с контролем доставки или TCP). Можно использовать протокол, аналогичный BitTorrent для больших файлов (piece selection). Поддержка докачки при обрыве.
Требования к обновлениям и стабильности
1. Сервис при запуске и затем периодически (раз в сутки) проверяет наличие новой версии на GitHub (или другом указанном репозитории). Сравнивает версию, скачивает, проверяет подпись и устанавливает.
2. Обновление происходит в фоне, без вмешательства пользователя (если не требуется перезапуск). При необходимости перезапуска сервис должен корректно завершить текущие соединения (отправить уведомления собеседникам о временной недоступности) и затем перезапуститься.
3. Crash reports: при аварийном завершении создается файл с логом, дампом, версией ОС, стеком. Сохраняется в папку логов. Пользователь может отправить его разработчику.
Требования к установке и автозапуску
1. На Windows: установщик (NSIS или аналоги) создает папку Program Files, добавляет службу Windows (или задание в планировщике для запуска от пользователя). Иконка в трее позволяет управлять сервисом.
2. На Linux: пакет .deb/.rpm или просто скрипт установки, копирующий бинарник в /usr/local/bin, создающий systemd user service (или системный). Включает автозапуск.
3. Сервис не требует прав администратора/root для работы (кроме установки). Все операции выполняются от имени текущего пользователя.
4. При установке создаются необходимые директории, генерируется конфигурация по умолчанию, генерируются ключи.
Требования к логированию
1. Логи пишутся в файл в папке данных (например, zibby.log). Логи ротируются (размер файла, количество).
2. Уровни логирования: error, warning, info, debug (настраивается). По умолчанию info.
3. Логи не должны содержать личную информацию (сообщения, ключи). Только системные события, ошибки, сетевые события (IP без портов? можно с портами, но без содержимого).
4. CLI может просматривать логи в реальном времени (tail) с фильтрацией.
Требования к производительности
1. Сервис должен поддерживать до 1000 одновременных соединений (в разумных пределах настольного ПК). Использование асинхронного ввода-вывода (Boost.Asio) с пулом потоков.
2. Шифрование/дешифрование не должно создавать значительных задержек. Используются быстрые алгоритмы (AES-GCM, ChaCha20-Poly1305, ECC для ключей).
3. Загрузка ЦП при бездействии – минимальна. При передаче больших файлов – загрузка ЦП до 30% (на современном оборудовании).
4. Оптимизация для работы на устройствах с ограниченными ресурсами (ноутбуки, Raspberry Pi) – режим экономии ресурсов.
Состав поставки
1. Исходный код на C++ (публичный репозиторий GitHub).
2. Сборка для Windows (exe, установщик). (УСТАНОВКА ДЛЯ ВСЕХ ПОЛЬЗОВАТЕЛЕЙ ЕСЛИ ЕСТЬ ПРАВА АДМИНА, ЕСЛИ НЕТ ТО НЕТ, НУЖНО ЗАПРАШИВАТЬ)
3. Сборка для Linux (бинарник, systemd unit, скрипт установки, deb).
4. Документация: руководство пользователя, руководство по сборке, описание API.
5. Примеры плагинов (если будет реализована система плагинов).
6. Crash report инструмент.
требования
1. Система плагинов: сервис может загружать динамические библиотеки (.dll/.so) из папки plugins. Плагины могут расширять функциональность (например, поддержка дополнительных протоколов, фильтры сообщений). Управление плагинами через CLI (включить/отключить).
2. Реестр и переменные пути: для сборки и использования из консоли должны быть установлены переменные окружения (или симлинки) для вызова zibby-service из любого места.
3. Поддержка локализации: сервис передает фронтендам информацию о языке системы и регионе, чтобы фронтенды могли адаптировать интерфейс. Сам сервис сообщения пользователю выводит на английском (для CLI можно сделать интернационализацию опционально).
